## Example: stored procedure with optimized query

The following shows a typical pattern for a parameterized stored procedure that retrieves paginated order data with a status filter.

```sql GetOrdersByStatus.sql
CREATE PROCEDURE dbo.GetOrdersByStatus
    @Status      NVARCHAR(50),
    @PageNumber  INT = 1,
    @PageSize    INT = 20
AS
BEGIN
    SET NOCOUNT ON;

    SELECT
        o.OrderId,
        o.CreatedAt,
        o.TotalAmount,
        o.Status,
        c.FullName   AS CustomerName,
        c.Email      AS CustomerEmail
    FROM dbo.Orders AS o
    INNER JOIN dbo.Customers AS c
        ON c.CustomerId = o.CustomerId
    WHERE o.Status = @Status
    ORDER BY o.CreatedAt DESC
    OFFSET (@PageNumber - 1) * @PageSize ROWS
    FETCH NEXT @PageSize ROWS ONLY;
END;
```

<Tip>
  Creating a composite index on `(Status, CreatedAt DESC)` for the `Orders` table lets SQL Server satisfy both the `WHERE` filter and `ORDER BY` clause in a single index scan, avoiding a costly sort operation.
</Tip>

## Data modeling approach

Gregori follows these principles when designing schemas:

- **Normalization** — at least third normal form (3NF) to eliminate redundancy and update anomalies
- **Referential integrity** — foreign key constraints enforced at the database level, not only in application code
- **Indexing strategy** — clustered indexes on primary keys, non-clustered indexes on frequently filtered and joined columns
- **Naming conventions** — consistent, descriptive names for tables, columns, and procedures that make the schema self-documenting

<Note>
  Stored procedures also provide a security boundary — callers can be granted `EXECUTE` permission on a procedure without having direct `SELECT` or `UPDATE` access to the underlying tables.
</Note>
